🔒 Agent 5 — Reference

Security Domains

The Security Agent reviews every codebase across up to 20 distinct security domains. Which domains run depends on the project's complexity tier — but you can always force all 20 with !security --full, or re-run a single domain after a fix with !security domain [N].

Domain Targeting Commands

CommandBehavior
!security Activate Security Agent; run the tier-appropriate domain set (Simple = 5 domains, Standard = critical+high domains, Complex = all 20)
!security --full Force all 20 domains regardless of complexity tier — use after large feature additions or for pre-production audit
!security domain [N] Re-run a specific domain only (e.g. !security domain 8 re-runs Dependency & Supply Chain after you've upgraded a vulnerable package). The domain number corresponds to the table below.
!security-review Run or route a targeted security review task outside the main gate flow
!security accept [ID] Formally accept a HIGH or MEDIUM finding with documented rationale — records the accepted risk in SECURITY_REPORT.md and HANDOFF v5

Review Tiers

The agent selects domains automatically based on the project complexity tier recorded in HANDOFF:

All 20 Security Domains

Domain 1

Defense in Depth

All Projects

Verifies that multiple independent security layers exist — no single failure can compromise the system.

Required Checks

  • Independent authentication layer (verify identity is checked separately from authorization)
  • Independent authorization layer (verify permissions are checked on every route/action, not just at login)
  • Input validation at API/IPC/native boundaries (not just client-side)
  • Rate limiting present and independent of application logic
  • Structured logging of auth, authorization, and data mutation events
  • Database-level permissions (app user has least-privilege access, no superuser in prod)
  • Release integrity layer (signed artifacts, dependency pinning)
Domain 2

Identity & Access Management

Projects with Users/Roles

Reviews the complete IAM surface — role model, permission matrix, and account lifecycle security.

Required Checks

  • MFA support present where risk warrants (admin accounts, payment-related actions)
  • Password policy: minimum length, complexity requirements, breach-check option (Have I Been Pwned API)
  • Password reset flow: secure token, short TTL (≤15 min), single-use, rate-limited
  • OAuth 2.0 flows use PKCE (Proof Key for Code Exchange) — not implicit flow
  • Role and permission model: roles map to discrete permission sets, no catch-all admin role in code
  • Horizontal access control: user A cannot access user B's resources by guessing IDs
  • Admin account separation: admin operations require explicit privilege, not role-checked implicitly
  • Account lockout or progressive delay after repeated failed login attempts
Domain 3

Authentication & Session Security

Auth Systems

Covers session lifecycle, token storage, and platform-specific auth credential security.

Required Checks — Web

  • Session cookies: HttpOnly, Secure, SameSite=Strict or Lax
  • JWT lifetime: short access token (≤15 min), refresh token rotation on use
  • Refresh token revocation path exists (database-backed, not stateless-only)
  • CSRF protection on all state-mutating endpoints (token or SameSite cookie)
  • No auth tokens stored in localStorage or sessionStorage (XSS exposure)

Required Checks — Native / Desktop

  • iOS: credentials stored in Keychain, not UserDefaults or plist
  • Android: credentials in EncryptedSharedPreferences or Android Keystore, not SharedPreferences
  • Windows: use Windows Credential Manager or DPAPI, not plain registry
  • macOS: use Keychain Services, not user defaults
  • Linux: use libsecret / system keyring, not plaintext config files
Domain 4

Network Security

All Networked Apps

Ensures all transport channels are encrypted and configured with correct security headers.

Required Checks

  • TLS 1.2+ enforced; TLS 1.0/1.1 disabled
  • HSTS header present (max-age ≥ 31536000, includeSubDomains)
  • Content-Security-Policy (CSP) header present and restrictive — no unsafe-inline or unsafe-eval except where documented
  • No mixed content (HTTP resources loaded on HTTPS pages)
  • iOS: App Transport Security (ATS) enabled — no NSAllowsArbitraryLoads
  • Android: android:usesCleartextTraffic="false" in manifest
  • Certificate validation not disabled in any environment (including dev/test stubs)
  • AbuseIPDB API check on any suspicious IP ranges in CORS allowlists or firewall rules
Domain 5

Input Validation & Injection

All Projects

Guards every trust boundary against injection attacks (SQL, XSS, command, path traversal, and more).

Required Checks

  • Server/API boundary validation on all incoming data (Zod, Joi, Yup, class-validator, or equivalent)
  • All database queries use parameterized queries or ORM-provided escaping — zero raw string interpolation
  • IPC/native bridge inputs validated before use on all desktop platforms
  • Output encoding: user-supplied content HTML-encoded before rendering (React's JSX handles this by default; custom dangerouslySetInnerHTML audited)
  • File upload: MIME type validated server-side; file extensions allowlisted; files stored outside web root
  • Path traversal: no user-supplied values used to construct file system paths without sanitization
  • No eval(), exec(), or child_process.exec() with user input without strict validation
  • Command injection: shell arguments quoted and escaped; prefer execFile() over exec()
Domain 6

Cryptography

All Projects with Secrets / Sensitive Data

Ensures only modern, audited cryptographic primitives are used throughout the codebase.

Required Checks

  • Passwords hashed with bcrypt (cost ≥ 12), Argon2id, or scrypt — never MD5, SHA1, or SHA256 alone
  • No custom cryptographic implementations — use vetted libraries only
  • JWT signed with RS256 or ES256 — never HS256 with a short or guessable secret
  • Encryption at rest: sensitive database fields encrypted; key stored outside the database
  • Secure randomness: use crypto.randomBytes(), SecureRandom, arc4random() — never Math.random() for security-critical values
  • No MD5 or SHA1 for integrity or MAC purposes (CRC/checksum uses are acceptable but must be labeled)
  • Deprecated algorithms: RC4, DES, 3DES — flag any use as Critical
Domain 7

Data Protection & PII

PII / Sensitive Data Projects

Validates that personal data is classified, protected, and handled according to applicable regulations.

Required Checks

  • PII classified in SCHEMA.md with explicit sensitivity labels (public, internal, sensitive, restricted)
  • Sensitive fields encrypted at rest (SSNs, payment card data, health records)
  • Logs redacted: PII fields not present in application logs; use structured log with field allow-list
  • Retention policy implemented: records deleted or anonymized after declared retention window
  • User data deletion path: GDPR right-to-erasure fulfilled, including cascades
  • GDPR/CCPA: lawful basis for processing documented; consent mechanism implemented where required
  • HIPAA: BAA established with all SaaS vendors who process PHI; PHI never logged
  • PCI DSS: card numbers not stored; use tokenization (Stripe, Adyen) for all card data
  • iOS: PrivacyInfo.xcprivacy present and accurate
  • Android: Data Safety form content matches actual data collection
  • DBHub (read-only): audit schema for unencrypted PII columns and overly broad row-level access
Domain 8

Dependency & Supply Chain

All Projects

The most automated domain — uses four external services to systematically check every dependency for known vulnerabilities, malware, and secret leakage.

Automated Dependency Intelligence Step (runs first)

  1. Extract dependencies from lockfiles: package-lock.json, yarn.lock, pnpm-lock.yaml, Cargo.lock, go.sum, Podfile.lock, build.gradle
  2. OSV.dev (free, always runs): POST each package+version to https://api.osv.dev/v1/query — covers npm, PyPI, Maven, Go, Rust, Swift. Collects all vulnerability IDs and severity scores.
  3. NIST NVD (free, always runs): Fetch CVE details for any OSV-identified vulnerabilities from https://services.nvd.nist.gov/rest/json/cves/2.0. Records CVSS v3 base score, attack vector, and fix version.
  4. GitGuardian (API key required): Full repo + git history scan. Detects secrets by type, location, and commit hash.
  5. VirusTotal (API key required): Hash-checks native binaries and pre-compiled artifacts.
  6. URLScan.io (API key required): Scans external API URLs for known malicious patterns and cert validity.

Required Manual Checks

  • Run platform-native audit: npm audit --audit-level=high, cargo audit, dotnet list package --vulnerable, Gradle dependency-check
  • Classify all CVEs by CVSS score: Critical (≥9.0) and High (7.0–8.9) block SHIP
  • Verify lockfiles are committed (no floating ^ ranges for production dependencies)
  • Check for abandoned packages: zero releases in 2+ years + no named maintainer = flag
  • GitHub MCP (read-only): inspect recent commits to key dependencies for supply-chain signs
Domain 9

Logging & Monitoring

All Production Systems

Validates that the application produces meaningful, tamper-resistant security-relevant events.

Required Checks

  • Authentication events logged: successful login, failed login (with IP/user-agent, without credentials), logout
  • Authorization failures logged: access denied events include resource, action, and identity
  • Data mutation events logged: create/update/delete of sensitive records include user identity and timestamp
  • Structured logging format (JSON or key-value) — not free-text strings
  • PII redaction: log pipeline strips or masks sensitive fields before persistence
  • Alerting: critical auth failures trigger an alert within 5 minutes (SIEM, PagerDuty, or equivalent)
  • Crash reporting configured (Sentry, Bugsnag, or equivalent) for production
  • Log integrity: logs forwarded to external system; local logs not the only copy
Domain 10

Error Handling

All Projects

Ensures that errors are safe — they inform the user without exposing internal state to attackers.

Required Checks

  • No stack traces returned to clients in production responses
  • No internal paths, library versions, or SQL query fragments in error messages
  • Consistent error response shape across all API endpoints (no leaking which fields failed validation beyond the safe message)
  • Unhandled promise rejections and uncaught exceptions caught globally and logged safely
  • 404 vs 403: returning 404 for unauthorized access is acceptable (avoids enumeration); returning 403 is also fine but never 500
  • External service errors (timeouts, downstream failures) handled with safe generic messages
Domain 11

Tenant Isolation

Multi-Tenant Projects

For SaaS products with multiple organizations or accounts — verifies every data access is scoped.

Required Checks

  • All database queries include a tenant_id, org_id, or account_id WHERE clause — no queries that can accidentally return cross-tenant data
  • Row-level security (RLS) enabled in Postgres where applicable — verified via DBHub read-only scan
  • Cross-tenant access test: manually attempt to access resource of org A using authenticated token of org B (should 403/404, never 200)
  • Admin impersonation (if present) requires explicit audit log entry and separate elevated permission
  • File/blob storage: paths include tenant scope; no path traversal to other tenants' files
  • API rate limits applied per-tenant, not per-IP (which is easily shared/spoofed)
Domain 12

Release Integrity & Updates

All Distributed Apps

Ensures distributed artifacts are signed and updates are delivered securely — preventing supply-chain attacks via the distribution channel.

Required Checks by Platform

  • iOS: Valid codesign, provisioning profile, App Store Connect archive
  • Android: APK/AAB signed with release keystore; keystore not committed to repo
  • Windows: Authenticode signing on installer and binaries (EV certificate for SmartScreen pass rate)
  • macOS: Hardened Runtime + notarization + stapling (Gatekeeper-compliant)
  • Linux: Package signing (GPG for APT/RPM; Flatpak verified bundle)
  • Desktop auto-update: Sparkle (macOS) or Tauri updater validate signature before applying; no unsigned update accepted
  • Web: Subresource Integrity (SRI) for any CDN-hosted scripts; package-lock.json committed
Domain 13

Client-Side Data & Secrets

All Projects

Prevents secrets from leaking into the client layer — browser bundles, mobile binaries, renderer processes, localStorage, and git history.

Required Checks

  • No API keys, tokens, or private keys in frontend JavaScript bundles (verified by grep + source map inspection)
  • No secrets in mobile app binary (decompile check for common patterns)
  • No secrets in Electron renderer process or Tauri WebView content
  • No sensitive data in localStorage or sessionStorage (XSS-accessible)
  • Source maps: disabled in production or served behind authentication — never public
  • No committed .env files with real values; .env in .gitignore
  • GitGuardian full repo + git history scan: any GitGuardian finding in this domain is automatically Critical severity — even if the secret was later removed from HEAD, it exists in git history and must be purged
🚨
Git history secrets are Critical

Removing a secret from HEAD does not remove it from git history. GitGuardian findings in git history require a force-push purge (git filter-repo) and key rotation — not just a new commit.

Domain 14

External Interface Surface

APIs / Webhooks / Deep Links

Audits every external-facing interface — public API endpoints, webhooks, deep links — for exposure and abuse vectors.

Required Checks

  • OpenAPI/Swagger documentation: all public endpoints documented; no undocumented endpoints reachable
  • Admin routes protected: /admin, /internal, /__debug routes require elevated auth and are not guessable
  • Webhooks: HMAC signature verification on all inbound webhook payloads (Stripe, GitHub, Twilio, etc.)
  • Deep links / OAuth redirect URIs: strict allowlist; no open redirect vulnerabilities
  • Unused endpoints removed: no legacy routes left active after feature deprecation
  • URLScan.io: submit all external API base URLs and webhook URLs for known malicious pattern check
  • GraphQL (if present): query depth limiting and query cost analysis enabled; introspection disabled in production
Domain 15

Security Test Cases

All Projects

Verifies that automated security tests exist — not just functional tests. Security test coverage is checked in the test suite.

Required Test Coverage

  • Auth bypass: test that unauthenticated requests to protected endpoints return 401
  • Horizontal escalation: test that user A cannot access user B's resources
  • Injection: test that SQL/XSS payloads are rejected or safely escaped
  • CSRF: test that state-mutating requests without CSRF token are rejected
  • Expired token rejection: test that expired JWT/session tokens return 401
  • Rate limiting: test that the rate limit triggers after N requests in T seconds
  • IPC/bridge tampering (desktop/mobile): test that native bridge inputs are validated before reaching native code
  • Input validation: test that invalid inputs (missing fields, wrong types, oversized payloads) return 400, not 500
Domain 16

Privacy & Compliance

Regulated / Privacy-Relevant Apps

Checks regulatory compliance requirements — GDPR, CCPA, HIPAA, PCI, COPPA, and platform privacy rules.

Required Checks

  • GDPR: lawful basis for each data category documented; consent banner present where required; data deletion endpoint implemented
  • CCPA: "Do Not Sell My Personal Information" link if selling data to third parties; opt-out mechanism
  • HIPAA: BAAs established with all vendors touching PHI; PHI not in logs; encryption in transit and at rest
  • PCI DSS: cardholder data not stored; tokenization used; compliance scope documented
  • COPPA: age verification present if service is accessible to users under 13
  • iOS: PrivacyInfo.xcprivacy manifest present; App Store privacy nutrition label accurate
  • Android: Data Safety form completed and accurate; declared permissions justified
  • Privacy policy: accessible without login; covers all data collection categories
Domain 17

Incident Response Readiness

Production Systems

Verifies that the team can respond to a security incident — not just prevent one.

Required Checks

  • Incident response runbook exists in docs/ or internal wiki: steps for common scenarios (data breach, credential compromise, DDoS)
  • Kill switch exists: ability to revoke all active sessions and rotate secrets within 15 minutes
  • Token/API key revocation path: each external API key can be rotated independently without downtime
  • Emergency contacts documented: security-specific contacts separate from general on-call
  • Vulnerability disclosure path: security.txt or /.well-known/security.txt present on web apps
  • Backup and restore verified: last restore tested within 30 days (or documented date)
Domain 18

Electron / Tauri IPC Security

Desktop Hybrid Apps

Desktop hybrid apps have a unique attack surface — the bridge between the web renderer and native OS. This domain covers both Electron and Tauri.

Electron Required Checks

  • nodeIntegration: false — no direct Node.js access from renderer
  • contextIsolation: true — renderer cannot access Electron internals
  • sandbox: true where possible
  • webSecurity: true (never disable, even in dev)
  • contextBridge exposes minimal surface — no broad Node.js APIs exposed
  • Zod/typed schema validation on all IPC messages before processing
  • CSP header set via session.setPermissionRequestHandler
  • Navigation validated: will-navigate and new-window events restrict to known origins
  • shell.openExternal() called only with allowlisted URLs

Tauri Required Checks

  • Minimal allowlist: only required capabilities enabled in tauri.conf.json
  • Restrictive CSP configured
  • All Tauri commands (Rust side) typed and validated — no raw string exec
  • No broad file system or shell access (fs.all, shell.all) without justification
Domain 19

React Native Bridge Security

React Native Mobile Apps

The React Native bridge is a privileged channel between the JS thread and native code — each native module is a potential attack surface.

Required Checks

  • Native modules expose minimal API — no passthrough to arbitrary OS calls
  • All inputs crossing the bridge validated on the native side (JS validation alone is insufficient)
  • WebView (if used): strict origin allowlist; onMessage handler validates sender origin before processing
  • Deep links: all deep link parameters validated and sanitized before routing
  • No sensitive data stored in AsyncStorage (unencrypted on device) — use react-native-encrypted-storage or equivalent
  • Third-party native modules audited: inspect source or published version for supply-chain risk
Domain 20

Node.js / Express API Security

Express / Node.js APIs

Express applications have a specific set of middleware requirements that, when missing, leave the API vulnerable to well-known attacks.

Required Checks

  • Helmet.js: configured and covering CSP, HSTS, X-Frame-Options, X-Content-Type-Options, Referrer-Policy
  • CORS: restricted to known origins — not origin: '*' in production
  • CSRF: csurf middleware or equivalent for session-based auth flows
  • Rate limiting: express-rate-limit on auth endpoints and public APIs
  • Body size limits: express.json({ limit: '10kb' }) or appropriate limit — prevents memory exhaustion
  • HPP (HTTP Parameter Pollution): hpp middleware present when query parameters are processed
  • Input validation: all route handlers use Zod/Joi/express-validator before reaching business logic
  • Auth/session: session secret is env-var-sourced, minimum 32 chars, regenerated on privilege elevation

Finding Format

Every finding in SECURITY_REPORT.md must use this exact structure:

## FINDING-NNN
**Severity:** Critical / High / Medium / Low
**Platform(s):** web / ios / android / desktop / all
**Domain:** N — [Domain Name]
**Evidence:** [exact file, line, or API response that demonstrates the issue]
**Exploit path:** [how an attacker could leverage this finding]
**Exact fix:** [specific code change or configuration update required]
**Acceptance criteria:** [how to verify the fix is complete]
**Status:** Open / Mitigated / Fixed / Accepted Risk

Launch Gate Decisions

DecisionMeaningCondition
APPROVE No blocking critical or high findings All critical findings resolved; all high findings either resolved or formally accepted via !security accept [ID]
CONDITIONAL Remaining risk is explicitly accepted and non-critical Only when documented mitigations exist and business risk is formally accepted by the user
BLOCK Pipeline cannot advance to Verifier Any unresolved Critical finding; exploitable auth/authz flaw; exposed secret in code or git history; unsigned release where signing is required; severe dependency CVE
