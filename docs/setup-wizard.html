⚙️ First-Time Setup

Setup Wizard

Before typing Start agent 0 for the first time, run the setup script. It walks you through two tiers of context management configuration and takes 2–3 minutes. Re-run it at any time to change your settings.

🚨
Required before pipeline activation

The Orchestrator reads config/context.json at activation. If setup has not been completed, it refuses to start and displays instructions to run the script.

Running the Setup Script

OSCommand
🐧 Linux / 🍎 macOS bash SETUP.sh
🪟 Windows Set-ExecutionPolicy -Scope Process -ExecutionPolicy Bypass
.\SETUP.ps1

Flags

FlagEffect
bash SETUP.sh (no flags)Full wizard — path personalisation then context configuration
bash SETUP.sh --checkShow current configuration and exit without making changes
bash SETUP.sh --resetClear all settings and re-run the full wizard from defaults

What Setup Does

  1. 1
    Path personalisation
    Replaces the default install path (/home/user/codesleuth/) with your actual install location throughout all pipeline files. Skipped automatically if paths already match. The resolved path is saved as install_path in config/context.json so agents have a single source of truth for the agents directory location.
  2. 2
    Tier 1 — local backend selection
    Three questions about coordination, semantic search, and agent memory. All Tier 1 options run locally with no accounts required. Current config is shown as the default — press ENTER to keep it.
  3. 3
    Tier 2 — cloud services (optional)
    Five optional cloud service prompts. Press ENTER to skip any you don't want. All Tier 2 services are additive — Tier 1 handles everything on its own.
  4. 4
    Package installation
    Installs any Python packages required by your Tier 1 selections (e.g. chromadb, mem0ai). Nothing is installed for Tier 1 options A (file-based defaults). Each package shows an animated progress bar so you can see install progress in real time. On systems where the system Python is externally managed (Debian, Ubuntu, and similar), setup automatically creates a .venv virtual environment in the agents directory and installs packages there — no manual intervention required.
  5. 5
    Connection tests
    Tests each configured backend. Reports ✅ or ❌ with a specific reason. Failures are reported but do not block saving — you can fix credentials and re-run.
  6. 6
    Config saved
    Writes config/context.json with all selected backends and a resolved active object. The Orchestrator reads this on every activation.

Tier 1 — Local Backends

No accounts. No internet. All options run on your machine.

Question 1 of 3 — Agent Coordination

Controls how parallel agents coordinate task status during builds.

OptionBackendBest for
A — File-based Plain files (default) Single-agent builds, simple projects
B — SQLite WAL ✨ recommended SQLite write-ahead log Parallel builds, larger projects — safer concurrent writes, no install required

Question 2 of 3 — Semantic Search

Controls how agents find relevant information inside large project files.

OptionBackendBest for
A — JSON index Keyword grep (default) Simple projects, low memory environments
B — Chroma local ✨ recommended Local vector database Large codebases — finds relevant context by meaning, not just keywords. Setup installs chromadb automatically.

Question 3 of 3 — Agent Memory

Controls how agents retain context between tasks and across sessions.

OptionBackendBest for
A — File-based HANDOFF.json (default) One-off projects, users who don't need cross-session memory
B — mem0 local ✨ recommended Local persistent memory store Ongoing projects — agents remember decisions, errors, and context across sessions. Setup installs mem0ai automatically.

Tier 2 — Cloud Services

All optional. Press ENTER at any prompt to skip. Tier 1 handles everything — Tier 2 adds visibility, scale, and team features.

Notion Cross-session project registry — every project, decision, error, and pattern stored in a searchable Notion database API key + database IDs notion.so/my-integrations
GitHub Issues Task graph mirror — each task becomes a GitHub Issue. Live project board for team visibility Personal Access Token + repo github.com/settings/tokens

Chroma Cloud Cloud-hosted semantic search — overrides local Chroma. Synced across machines API key trychroma.com

Configuration File

Setup writes all choices to config/context.json in the agents directory. The active object at the bottom is the resolved backend map the Orchestrator reads — Tier 2 services automatically override Tier 1 when configured.

💡
Automatic virtual environment

On Debian, Ubuntu, and similar systems the system Python is externally managed (PEP 668) and blocks pip install by default. Setup detects this automatically and creates a .venv virtual environment in the agents directory, then installs packages there. The resolved path is recorded in active.venv so agents know where to find the packages — no manual source .venv/bin/activate step is required. If active.venv is null, packages are installed into the system Python directly.

💡
registered_projects — always-resident pipeline

The registered_projects map is written automatically by Agent 3 when a build completes, and updated by Agents 4–6 as the pipeline advances. On every subsequent Start agent 0, the Orchestrator reads this map and displays a resume menu — so the pipeline is always resident for any registered project, even across sessions. You never need to re-activate the pipeline for a project you have already built. See Maintenance Mode for the always-resident mini-pipeline that handles post-build features and fixes.

💡
Override resolution

When a Tier 2 service is enabled it automatically overrides the corresponding Tier 1 backend in the active object — no manual editing required. For example: enabling mem0 Cloud sets active.agent_memory = "mem0_cloud" regardless of the Tier 1 memory choice.

What Should I Choose?

You are…Recommended config
New to CodeSleuth, not technical Tier 1 all B (SQLite + Chroma local + mem0 local). Skip all Tier 2. Best balance of capability and zero friction.
Developer, single machine Tier 1 all B. Add mem0 Cloud and Chroma Cloud from Tier 2 when you want cross-machine sync.
Developer, working in a team Tier 1 all B. Add Redis + GitHub Issues from Tier 2 for real-time coordination and shared project visibility.
Power user / enterprise Tier 1 all B. Enable all Tier 2 services for maximum persistence, visibility, and scale.
