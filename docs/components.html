🧩 UI Components

Component Libraries

CodeSleuth ships with two pre-integrated component libraries: Boron (a static HTML/Bootstrap admin template with 100+ pages) and Materialize (a Material Design component framework). Agents auto-discover these libraries at Planning activation and incorporate them into TDD Section 5A and COMPONENT_MAP.md.

Auto-Discovery

The Planner (Agent 2) automatically discovers component catalogs by scanning components/**/COMPONENT_CATALOG.md at activation. Any COMPONENT_CATALOG.md file found anywhere under components/ — at any depth — is loaded and merged into the combined component inventory.

💡
No pipeline changes required

A new library dropped into components/ with a COMPONENT_CATALOG.md is automatically included in planning with no pipeline changes required. The Planner reads it, the Builder uses it, the Verifier confirms it.

Boron — Admin Template Library

Boron provides HTML/CSS layout patterns and UI blueprints sourced from a Bootstrap-based admin template. It ships with 100+ pre-built HTML pages in components/Boron/dist/.

How agents use Boron

Boron is a structural reference, not a framework dependency. Agents translate Boron patterns to the target platform's component system:

PlatformHow to use Boron patterns
Next.js / React Translate HTML structure to JSX. Use MUI or shadcn/ui for interactive elements. Apply Tailwind classes for spacing/layout.
Vue.js Convert to Vue SFC templates. Replace class-based JS with Vue Composition API.
Angular Convert to Angular templates with @Component. Replace jQuery with Angular directives.
Svelte Convert to Svelte components. Use <script> for reactive state.
Vanilla JS / HTML Use directly from dist/ HTML files. Adapt CSS as needed.

Component Library Resolution

When multiple libraries exist or a user preference conflicts with the stack default, the Planner runs the Component Library Resolution Step before any artifact is written:

SituationResolution
No catalog exists, no user preference Use the Platform Stack Default (e.g. shadcn/ui for Next.js)
No catalog exists, user stated a preference Use stated preference
Catalog exists, matches stack default Use catalog
Catalog exists, no conflict with stack default Use catalog; log choice in DECISIONS.md
Catalog exists AND conflicts with stack default Surface conflict to user — pause and wait for Option A or B confirmation

Conflict output format

⚠️ Component library conflict detected.

Existing catalog: Materialize — 42 components, 0 stories, framework: CSS/JS
Platform default: shadcn/ui + Tailwind CSS

Option A — Use existing catalog: Materialize
  Source: components/Materialize/next-version/COMPONENT_CATALOG.md
  Benefit: 42 components ready; consistent Material Design across app
  Risk: Bundle size; requires Materialize CSS; no Storybook stories yet

Option B — Use platform default: shadcn/ui
  Benefit: Tailwind-only, smaller footprint, matches Next.js stack default
  Risk: 42 components in existing catalog would not be used; starting from zero stories

Type A or B to continue planning.

COMPONENT_MAP.md

When a components/ folder exists, the Planner generates artifacts/build/COMPONENT_MAP.md — a dependency map linking pages to their required components:

## Component Dependency Map

| Page / Feature    | Components Used                    | Story Exists | Task (if story missing) |
|-------------------|------------------------------------|--------------|-------------------------|
| Booking page      | BookingCalendar, Button, FormField  | ❌ BookingCalendar | TASK-005 |
| Pricing page      | PricingCard, Button                | ✅ all       | —                       |
| Dashboard         | DashboardMetric, ProgressChart     | ❌ both      | TASK-006, TASK-007      |

Rules enforced by the Builder:

  • Before writing any new UI element, check COMPONENT_MAP.md and COMPONENT_CATALOG.md first
  • If an equivalent component exists in the catalog, use it — never create a duplicate
  • A genuinely new component must be introduced via a component task (story first, then implementation) before it can be used in a page task
  • No page or feature task may reference a component that lacks at least one story

Storybook Integration

When Storybook is in scope, component tasks follow a dedicated workflow:

  1. Read the catalog entry for the component from COMPONENT_CATALOG.md
  2. Write stories first — create .stories.tsx files with real prop values (no placeholder text, no empty renders)
  3. Run pnpm build-storybook — must exit zero
  4. Write visual baseline: pnpm test:visual -- --update-snapshots targeting the component's spec file
  5. Implement the component (if it doesn't yet exist) or patch it if story rendering exposed a gap
  6. Re-run pnpm build-storybook and pnpm test:visual — both must exit zero

Story naming convention

// File: src/components/BookingCalendar/BookingCalendar.empty.stories.tsx
// File: src/components/BookingCalendar/BookingCalendar.withSlots.stories.tsx
// File: src/components/BookingCalendar/BookingCalendar.error.stories.tsx

// Pattern: ComponentName.variant.stories.tsx
