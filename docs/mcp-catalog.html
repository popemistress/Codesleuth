🔌 MCP Servers

MCP Server Catalog

CodeSleuth uses Model Context Protocol (MCP) servers to give agents live access to external tools — databases, browsers, APIs, and more. Each agent is assigned an approved risk tier, and no agent may use a server above its tier without explicit user confirmation.

Catalog Location

The active MCP catalog is stored at artifacts/pipeline/MCP_CATALOG.md in the agents directory — not the project directory. This is the single source of truth for which MCP servers are approved for each agent.

Risk Tiers

TierDescriptionExamples
Low Read-only access to public data; no credentials required; no write capability Brave Search, Context7, Hacker News API, npm Registry
Medium Read-write access with limited scope; API key required; container-isolated preferred Firecrawl (read), Redis, Neon (branch-based), Playwright
High Write access to persistent data; token required; branch restrictions enforced GitHub (write), DBHub (schema), Supabase
Critical Broad system access; explicit user confirmation required before every use Shell access, file system without scope restriction

Catalog Management Commands

CommandBehavior
!mcp-catalog Display the active catalog with risk levels, capabilities, and agent assignments
!mcp-add [server] Scan the server with cisco-ai-defense/skill-scanner, classify risk, add to catalog on pass. The scan checks for malicious code, prompt injection, and supply-chain risks before any server is approved.
!mcp-disable [server] Remove server from active use without deleting its catalog entry. The entry is preserved with a Status: Disabled marker for future reference.

MCP Safety Rules

  1. Skill scanner required: Before any new skill package or MCP server is used by the pipeline, cisco-ai-defense/skill-scanner must run against it to check for malicious code, prompt injection, and supply-chain risks.
  2. Risk tier enforcement: No agent may use a server above its approved tier without explicit user confirmation typed as a standalone command.
  3. Docker preference: Prefer Dockerized MCP servers (mcp.docker.com catalog) for local deployments — container isolation prevents host pollution and simplifies removal.
  4. SaaS gateway: For projects requiring many SaaS integrations (Slack, Jira, Google Drive, Notion), route through the Composio managed MCP gateway (composio.dev/mcp) rather than requiring per-service OAuth setup from the user.
  5. GitHub write restriction: Builder may only push to the project's feature branch. Never push directly to main, master, or release branches. Require user confirmation before any force-push or branch deletion.
