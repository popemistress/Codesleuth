⚡ Multi-Agent Workloads

Parallel Execution

CodeSleuth runs multiple tasks and even multiple pipeline branches concurrently. This page covers every mechanism: the goroutine runner for shell-level parallelism, sub-agent spawning for agent-level parallelism, and session forking for architecture-level parallelism.

Three Levels of Parallelism

LevelMechanismScopeWhen used
Shell Go goroutine runner (parallel_runner.go) Multiple build tasks within one pipeline branch Parallel batches in TASK-GRAPH Parallelism Map
Agent Sub-agent spawning via Claude Code Agent tool Multiple tasks with full agent context Architecture-class tasks that benefit from isolated context
Pipeline Session forking (session_fork.py) Two complete pipeline branches exploring different architectures !fork [description] — exploring alternative approaches

Reading the Parallelism Map

The Planner (Agent 2) writes a Parallelism Map at the end of TASK-GRAPH.md. The Builder reads this map at activation to know which tasks can run concurrently:

## Parallelism Map
Batch 1 (no dependencies): [TASK-001]
Batch 2 (depends on Batch 1): [TASK-002, TASK-003, TASK-004]
Batch 3 (depends on Batch 2): [TASK-005, TASK-006]
Batch 4 (depends on Batch 3): [TASK-007]
  • Batch with 1 task → run sequentially (standard Per-Task Cycle)
  • Batch with 2+ tasks → run in parallel using Method A or Method B below
  • Tasks within a batch must not write the same output file — the Planner ensures this via can_parallel: true/false on each task

Task Dependency Annotations

Every task in TASK-GRAPH carries these fields to enable correct parallel scheduling:

FieldValuesPurpose
task_typeapi | auth | data | infra | ui | config | feature | cicd | obsPattern Library matching and domain routing
depends_onTASK-NNN list or "none"Declares prerequisite tasks — batch placement
can_paralleltrue | falseWhether this task can run alongside siblings in the same batch
output_filesList of file pathsConflict detection — two tasks writing the same file must be sequential

can_parallel: false when: the task writes files also written by another task in the same batch, or the task requires reading output produced by another task in the same batch.

Method A — Go Goroutine Runner

Preferred for shell-level parallelism. The Go runner spawns each task as a goroutine, detects write conflicts, and writes structured results to JSON.

How to invoke

# Write batch config
cat > /tmp/batch.json << 'EOF'
{
  "tasks": ["TASK-003", "TASK-004", "TASK-005"],
  "cmd_template": "python /path/to/agents/scripts/run_task_worker.py --task {task_id} --project-dir {project_dir}",
  "project_dir": "/path/to/project",
  "timeout_seconds": 300
}
EOF

# Run all tasks concurrently with goroutines
go run /home/user/codesleuth/scripts/parallel_runner.go \
    -batch-file /tmp/batch.json \
    -task-graph artifacts/build/TASK-GRAPH.md \
    -output artifacts/telemetry/batch_results.json

Conflict detection (run before the batch)

go run /home/user/codesleuth/scripts/parallel_runner.go \
    -tasks TASK-003,TASK-004,TASK-005 \
    -task-graph artifacts/build/TASK-GRAPH.md \
    -cmd "echo {task_id}" \
    -output /dev/null 2>&1 | grep "conflict"

Any reported conflict means those tasks must be split into separate sequential batches. The runner handles this automatically — conflicting tasks are moved into sub-batches and run sequentially within the batch, while non-conflicting tasks still run in parallel.

Output

After completion, read artifacts/telemetry/batch_results.json and mark each task's status in TASK-GRAPH.md accordingly. Each task result carries:

{
  "task_id": "TASK-003",
  "status": "done" | "blocked" | "conflict",
  "exit_code": 0,
  "stdout": "...",
  "stderr": "...",
  "duration_seconds": 42
}

Method B — Parallel Sub-Agents

Preferred for agent-level parallelism — when each task needs full model reasoning, INTERFACES.md context, or complex multi-file implementation. Sub-agents run completely concurrently because all Agent tool calls appear in the same response.

How to invoke

Agent(description="Implement TASK-003: Auth module",  prompt="...")
Agent(description="Implement TASK-004: User API",     prompt="...")
Agent(description="Implement TASK-005: Booking form", prompt="...")
💡
All Agent calls in a single response

Claude Code executes multiple Agent tool calls in a single response concurrently. If Agent calls appear in separate responses, they run sequentially. The Builder always groups parallel batch Agent calls into one response.

Sub-agent prompt requirements

Each sub-agent prompt must include — because the sub-agent starts with no session context:

  • The full task entry from TASK-GRAPH.md
  • The relevant INTERFACES.md section for this task's module
  • The relevant SCHEMA.md section (if the task touches data)
  • Instructions to record results in TASK-GRAPH.md
  • Instructions to append a telemetry record to artifacts/build/WRITE_LOG.jsonl
  • The project directory path (to enforce the agents directory isolation rule)

After sub-agents complete

  1. Collect all sub-agent results
  2. Run SAST scan on every changed file: python scripts/sast_scan.py <file> --task TASK-NNN
  3. Run coverage check for each task: python scripts/check_coverage.py --task TASK-NNN
  4. Run Change Impact Check: cross-reference modified files against FILE_OWNERSHIP_MAP.md
  5. Append telemetry records to artifacts/telemetry/tasks.jsonl

Session Forking — Pipeline-Level Parallelism

The !fork command creates two isolated pipeline branches from the current project state. Branch A continues with the existing plan; Branch B explores an alternative approach. Both run concurrently as separate agent processes.

When to use !fork

  • Undecided between two architectures (e.g., PostgreSQL vs SQLite, REST vs GraphQL)
  • Multi-platform projects where each platform needs its own pipeline run
  • Exploring a risky refactor without committing to it before seeing the result

Step 1 — Create the fork

# The Orchestrator runs this automatically when !fork is typed
python /home/user/codesleuth/scripts/session_fork.py create \
    --project-dir [project_dir] \
    --session-file artifacts/build/FORK_SESSION.json \
    --branch-b-desc "use PostgreSQL instead of SQLite"

The script creates [project]-fork-a/ and [project]-fork-b/ as isolated copies of the project directory. HANDOFF.json is present in both.

Step 2 — Run both forks concurrently

The Orchestrator spawns two Agent calls in a single response:

Agent(
  description="Fork-A pipeline run: original SQLite approach",
  prompt="You are Agent 3 (Builder) operating in project directory [fork-a-dir].
          HANDOFF.json is already present. Run the full pipeline from the current
          state using the original plan. Record pipeline_verdict in HANDOFF.json
          when complete. Do not interact with the user — run to completion."
)
Agent(
  description="Fork-B pipeline run: PostgreSQL alternative",
  prompt="You are Agent 3 (Builder) operating in project directory [fork-b-dir].
          HANDOFF.json is already present. Run the full pipeline using this
          alternative approach: use PostgreSQL instead of SQLite.
          Record pipeline_verdict in HANDOFF.json when complete."
)

Step 3 — Compare results

python /home/user/codesleuth/scripts/session_fork.py compare \
    --session-file artifacts/build/FORK_SESSION.json

The comparison report shows: build success rate, test coverage, SHIP/NO-SHIP verdict, task durations, and open blockers for each fork side by side.

Step 4 — Merge the winner

# After the user types !merge fork-a or !merge fork-b
python /home/user/codesleuth/scripts/session_fork.py merge fork-a \
    --session-file artifacts/build/FORK_SESSION.json

The script copies the winning fork's project directory back to the canonical project path, commits the merge to git, removes both fork directories, and archives the session file.

⚠️
!fork vs !nofork

When the Orchestrator detects a multi-platform project or an undecided architecture choice, it presents a fork plan and waits. Type !fork to approve and start both branches, or !nofork to decline and proceed with a single branch.

Pipeline Replay — Point-in-Time Restore

!replay TASK-NNN is the recovery mechanism for bad task outcomes — it restores the codebase to the git commit created after TASK-NNN completed, marks all subsequent tasks PENDING, and resumes the Builder from the next task.

When to use !replay

  • A task produced incorrect output and all earlier work was correct
  • A !change event requires rebuilding from a known-good checkpoint
  • Trying a different implementation strategy from a specific point
💡
Checkpoints require commit_by_task.py

Replay works because commit_by_task.py records one git commit per completed task. Each task's commit hash is written to TASK-GRAPH.md under a **Checkpoint:** field. If no checkpoints exist, replay falls back to searching git log for a commit referencing the task ID.

How to invoke

python /home/user/codesleuth/scripts/pipeline_replay.py \
    --task TASK-003 \
    --task-graph artifacts/build/TASK-GRAPH.md \
    --project-dir [project directory]

Output

✅ Replay ready
   Codebase:    restored to state after TASK-003 (commit abc1234)
   Tasks reset: 4 task(s) marked PENDING
   Resume from: TASK-004

   Type !build to resume.

Build Progress Display

During parallel and sequential builds, the Builder outputs a two-line status after every task — a status line and a progress bar. The bar is always 20 characters wide.

⚠️
Precondition: Builder must be active

This display only appears when Agent 3 (Builder) is running tasks — meaning the pipeline has been activated (Start agent 0) and has advanced through Discovery → Planning → !build. Asking Claude to build something directly without going through pipeline activation produces no progress display.

The format is embedded directly in step 15 of the Builder's Per-Task Cycle so it fires after every task:

--- Task 5: Auth module → DONE ✅ | TC: 4/4 pass | Regression: ✅ 2 tasks checked | Gates: ✅ lint ✅ types ✅ tests ✅ build
    ████████████░░░░░░░░  12 / 24 tasks complete  ·  50%

--- Task 6: User API → DONE ✅ | TC: EXEMPT (config) | Gates: ✅ lint ✅ build
    ███░░░░░░░░░░░░░░░░░   3 / 24 tasks complete  ·  12%

--- Task 7: Booking form → DONE ✅ | TC: 5/5 pass (3 visual) | Baselines: created | Gates: ✅ visual ✅ build
    ████████████████████  24 / 24 tasks complete  · 100% 🎉

Progress bar rules

  • Bar width: always 20 characters. Filled: █. Empty: ░
  • Formula: filled = round(done / total × 20)
  • done = tasks with status DONE or DONE_WITH_CONCERNS
  • BLOCKED and STALE tasks are not counted as done
  • Dead-lettered (skipped) tasks count toward total but not done
  • At 100%, appends 🎉

TC status codes in the status line

CodeMeaning
TC: N/N passN test contracts confirmed failing before implementation; all N passing after
TC: EXEMPT (category)Task was in a valid exempt category (scaffold, install, config, types, migration, ci)
TC: DERIVED N/N passBuilder derived the contract from acceptance criteria because Planner omitted it
TC: N/N pass (N visual)Includes visual regression tests — Playwright screenshot comparisons

Staged Write Overlay

Every file write in the pipeline goes through an approval gate before hitting disk. This prevents accidental overwrites, secret leakage, and incorrect implementations from persisting.

The write cycle for every file

  1. Secret scan: python scripts/secret_scan.py <proposed_content> — blocks on any detected secret pattern
  2. Stage the write: python scripts/staged_write.py stage --task TASK-NNN --file <path> --content <temp_file> --rationale "..."
  3. Approval result: exit 0 (applied) → proceed; exit 1 (rejected) → re-enter implementation with rejection reason
  4. Hot compile check: immediately after every file write, run the compile command (go build ./..., npx tsc --noEmit, cargo check, etc.)
  5. Write log record: python scripts/write_log.py record --task TASK-NNN --file <path>
  6. SAST scan: python scripts/sast_scan.py <file> --task TASK-NNN (non-blocking)
💡
CI mode: --auto-apply

In non-interactive CI environments, pass --auto-apply to staged_write.py to skip the prompt. The secret scan still runs — staged_write is the approval gate, not the security gate.

RARV Cycle — Per-Task Test Loop

Every non-exempt task follows the RARV cycle: Read → Assert → Repair → Verify. This is the inner loop that catches implementation errors before the full gate suite runs.

  1. Read — Load task context: CHECKPOINT.md, task entry, INTERFACES.md section, SCHEMA.md section, owned source files
  2. Assert — Write failing tests first. Run the targeted test command. Confirm tests exit non-zero before writing any implementation.
  3. Repair loop — Implement the minimum working change. Run scoped tests. If fail → repair (up to 3 self-repair attempts, then Repair Mode).
  4. Verify — Run the full targeted gate suite. Coverage check. SAST scan. Visual baseline (UI tasks). Change Impact Check.
