🔧 Agent 7

Maintenance Mode

Agent 7 is the always-resident pipeline for projects that have completed a build. Every feature addition, bug fix, and change — however small — passes through a lightweight mini-pipeline rather than being applied ad hoc. The pipeline never goes dark for a registered project.

🚨
Ad-hoc edits are refused

Once a project is registered, direct edits outside the pipeline are blocked. Every change — one line or one thousand — must enter the mini-pipeline via Agent 7. This is intentional: it keeps the HANDOFF, TASK-GRAPH, test contracts, and CHANGE_LOG in sync with the actual codebase at all times.

How Projects Become Registered

Registration is automatic — you do not trigger it manually. Agent 3 (Builder) calls register_project.py at the moment all tasks are marked DONE, before activating the Critic. This means a project is registered even if the user stops at the completed build and never proceeds to Critic, Security, or Verifier.

The Mini-Pipeline

Every change request runs through a six-stage cycle. The cycle is lighter than the full pipeline — no TDD rewrite, no full planning ceremony — but all safety gates are preserved.

  1. 1
    Scope (Mini-Discovery)
    Agent 7 produces a scope block: type, surface, risk level, likely files touched, and whether an automatic security check will run. A MAINT-NNN entry is appended to CHANGE_LOG.md.
  2. 2
    Plan (1–5 tasks)
    A compact task table is produced — no full TDD, no INTERFACES rewrite. Each task has a test contract (REQUIRED or EXEMPT), gate commands, and owner files. Agent 7 waits for !m-build before proceeding.
  3. 3
    Build
    Same per-task rules as Agent 3: failing tests first, minimum implementation, platform gates, SAST scan, secret scan before every write. Up to 5 self-repair attempts per task before surfacing a blocker.
  4. 4
    Critic check (automatic, targeted)
    Three focused questions scoped to the changed surface only: Does this degrade an existing user flow? Does it introduce UX inconsistency? Does it miss an obvious edge case? Concerns are surfaced before commit; all-clear proceeds automatically.
  5. 5
    Security (automatic, surface-gated)
    Runs automatically only when the change surface is auth, data, or infra. Scoped to the diff — not the full codebase. HIGH findings block commit. All other surfaces skip this stage and proceed directly to Stage 6.
  6. 6
    Commit & registration update
    commit_by_task.py groups writes by task ID and creates conventional commits. register_project.py advances the stage to maintenance. The cycle-complete banner is displayed and Agent 7 returns to ready state — no re-activation needed for the next request.

Maintenance Commands

CommandBehavior
!m-build Human gate — advances from Plan to Build within the current change cycle.
!m-status Shows current cycle ID, scope, open tasks, last commit, and registration stage.
!m-skip Discard the current change cycle without committing. Returns to ready state.
!m-history List all MAINT-NNN cycles applied to this project with type, surface, and commit reference.
!observe Read-only mode — loads project context but refuses all writes. Use when inspecting the codebase without intending to change it. Type !observe off to exit.
💡
Standard pipeline commands still work

!retry, !skip, !change, !error-check, !pattern-add, !telemetry, and !docs are all available during an active maintenance cycle.

Observe Mode

Sometimes you want to inspect a registered project — read code, check the HANDOFF, review the TASK-GRAPH — without triggering a change cycle. !observe activates a read-only session: 

👁 Observe mode active — reads only, no writes. Type !observe off to resume.

Agent 7 will load files, answer questions, and summarize the codebase, but refuses all tool writes, plan execution, and gate commands until !observe off is issued.

Under the Hood — register_project.py

The always-resident behavior is backed by scripts/register_project.py, which writes to the registered_projects object in config/context.json. Each entry records: 

{
  "/home/user/myapp": {
    "name":              "MyApp",
    "status":            "maintenance",
    "pipeline_stage":    "maintenance",
    "last_active_agent": 7,
    "handoff":           "/home/user/myapp/artifacts/build/HANDOFF.json",
    "registered_at":     "2026-05-31T21:00:00Z",
    "last_updated":      "2026-06-01T09:14:00Z"
  }
}

Stage progression is one-way — the script refuses to regress a stage. Calling it with an earlier stage than the current value produces a warning and leaves the entry unchanged.
