📁 Artifact Schemas

Artifacts Reference

Every artifact produced by the CodeSleuth pipeline has a defined path, producer, consumer, and schema. This page is the complete reference — from Discovery's first feature-spec.md to the Verifier's final VERIFICATION_REPORT.md.

🚨
Two distinct artifact namespaces

Pipeline artifacts (artifacts/pipeline/) live in the agents directory and persist across all projects. Project artifacts (artifacts/discovery/, artifacts/build/, etc.) live in the project directory. These two namespaces must never be conflated — writing pipeline artifacts to the project directory or vice versa is a hard violation.

Pipeline Artifacts (agents directory)

Path: /home/user/codesleuth/artifacts/pipeline/ — shared across all projects.

FileProducerConsumerContents
KNOWN_ERRORS.md Orchestrator (!error-add) All agents (repair loops) Cross-project error registry. ERROR-NNN entries with symptom, root cause, fix, guardrail, detection, and stack.
KNOWN_ERRORS_INDEX.md Orchestrator (!error-add) All agents (grep lookup only) Keyword → Error ID mapping. One row per keyword. Never read KNOWN_ERRORS.md directly — always grep this index first.
PATTERN_LIBRARY.md Orchestrator (!pattern-add) Planner (activation), Builder (architecture tasks) Successful implementation patterns. PATTERN-NNN entries with domain, stack, context, pattern, pitfall avoided, reuse signal, and evidence.
PATTERN_INDEX.md Orchestrator (!pattern-add) Planner and Builder (grep lookup only) Keyword → Pattern ID mapping. Never read PATTERN_LIBRARY.md in full — always grep this index first.
PIPELINE_IMPROVEMENTS.md pipeline_improve.py (post-SHIP) Orchestrator (!pipeline-health) Auto-generated improvement proposals from telemetry analysis: slowest tasks, most-repaired tasks, recurring gate failures.
MCP_CATALOG.md Orchestrator (!mcp-add) All agents (tool selection) Approved MCP servers with risk tier, capabilities, auth requirements, and agent assignments.
DEAD_LETTER.md Orchestrator (!skip) Verifier (ship readiness check) Skipped tasks with: task ID, reason skipped, downstream impact, recovery path, and whether it blocks SHIP.

Discovery Artifacts (project directory)

Path: [project]/artifacts/discovery/ — written by Agent 1, read by Agent 2.

FileTriggerContents
feature-spec.md !spec 22-section product specification: problem, users, core features, user stories, flows, data requirements, platform scope, native capabilities, UX, API/business logic, auth/security, roles/permissions, store targets, monetization, i18n, performance, observability, scope boundaries, risks, definition of done, open questions, recommended integrations.
HANDOFF.json (v1) !spec Structured discovery state: spec_version, pipeline_stage, project_name, platform_scope, web_stack, design_contract, complexity_tier, entity_model, api_contracts, native_capabilities, auth_model, feature_parity_matrix, store_targets, monetization, i18n_scope, performance_targets, market_validation, capability_intelligence.
PRE_SCAN.md Agent 1 activation Autonomous pre-scan output: codebase scan (greenfield/existing), Pattern Library matches, KNOWN_ERRORS warnings, dependency research findings, risk register with complexity tier estimate.

Build Artifacts (project directory)

Path: [project]/artifacts/build/ — written by Agent 2 (Phase A+B) and updated by Agent 3.

FilePhaseProducerContents
TDD.md Phase B Agent 2 21-section Technical Design Document: platform scope matrix, repository structure, data layer, API/interface layer, auth architecture, UI architecture per platform, web design contract (Section 5A), native capabilities, caching, i18n, error handling, feature parity, testing architecture, security requirements, CI/CD plan, code signing, store submission, performance targets, rollback strategy, known limitations, change log.
INTERFACES.md Phase B Agent 2 Module interface contracts: for every module — path, platform, owner task, exported types, exported functions with exact signatures, middleware/constants/events, cross-module dependencies, changelog.
contracts/ Phase B Agent 2 Machine-readable contract layer: Zod schema files ([module].contracts.ts) and OpenAPI stub (openapi.yaml). Builder validates code against these; Verifier runs contract conformance tests against them.
SCHEMA.md Phase B Agent 2 Data model: entities, fields with types/nullability/defaults, relationships, indexes, constraints, migration strategy, seed data (if needed), sync/offline metadata for native platforms.
TASK-GRAPH.md Phase B Agent 2 Complete task dependency graph: for every task — TASK-NNN, status, task_type, platform, dependencies, can_parallel, output_files, acceptance criteria, test contract status (REQUIRED/EXEMPT), test contract table (behavior descriptions), gate commands, checkpoint hash. Includes Parallelism Map at end.
FILE_OWNERSHIP_MAP.md Phase B Agent 2 Maps every planned source file to the task(s) that own it. Used by Builder for Change Impact Tracking — after each task, cross-reference modified files against this map.
COMPONENT_MAP.md Phase B Agent 2 Component dependency map: page/feature → components used → story status → task (if story missing). Only present when components/ folder exists.
DECISIONS.md Phase A + B Agent 2 (+ Agent 3 updates) Architecture and implementation decision log: each decision with context, options considered, decision made, rationale, and consequences. Patterns applied are recorded here as PATTERN-NNN: applied.
CHECKPOINT.md Phase B (+ running updates) Agent 2 (+ Agent 3) Builder progress state: current task, completed tasks, open blockers, test contract summary (REQUIRED/confirmed-failing/derived/exempt counts). Updated every 3–5 tasks or after a major blocker.
SYMBOL_INDEX.md Phase B Agent 2 (ctags) Codebase symbol map: file path → exported symbols → signatures. Built by ctags at Planner activation. Tasks annotated with reuse_existing where applicable.
HANDOFF_v2_staged.json Phase B Agent 2 Staged v2 HANDOFF (promoted to HANDOFF.json by Builder at activation): all v1 fields plus schema_version v2, monorepo strategy, stack_versions, component_library, symbol_index_path, open blockers.
HANDOFF.json (v3) Build completion Agent 3 v3 HANDOFF with build status, feature parity, auth model, store targets, open blockers, gate results, coverage_pct.
WRITE_LOG.jsonl Per file write Agent 3 (write_log.py) Newline-delimited JSON log of every file write: task_id, file_path, timestamp. Consumed by commit_by_task.py post-SHIP to group git commits by task.
COVERAGE_LOG.md Per task completion Agent 3 (check_coverage.py) Code coverage history: one row per task with coverage percentage. Last row shows overall baseline at build completion.
CHANGE_LOG.md On !change-approve Orchestrator Spec change history: CHANGE-NNN entries with description, stale tasks, invalidated contracts, schema delta, re-plan scope.
CONTEXT_ARCHIVE.md On context compression Orchestrator Archive of full conversation exchanges for compressed tasks. Content is moved here — never deleted. Append-only.
FORK_SESSION.json On !fork Orchestrator (session_fork.py) Fork session state: branch A and B descriptions, project directory copies, SHIP/NO-SHIP verdicts after completion.
SEMANTIC_INDEX.json After scaffold task Agent 3 (semantic_search.py) Semantic search index of codebase chunks. Updated after every file write. Used by Builder before architecture-class tasks to find existing implementations.
phases.md Phase B Agent 2 Phased implementation plan: tasks grouped by phase with lint/type/test/build result columns and TC status (N/N, EXEMPT, DERIVED N/N, BLOCKED).

Security Artifacts (project directory)

Path: [project]/artifacts/security/ — written during build (ongoing) and security review.

FileProducerContents
SAST_FINDINGS.jsonl Agent 3 (sast_scan.py) — after every file write Static analysis findings: one JSON line per finding with file, line, rule, severity, message, task_id. Agent 5 reads this at activation and must confirm, expand, or dismiss every finding.
SECRET_BLOCKS.jsonl Agent 3 (secret_scan.py) — on detection Blocked secret writes: task_id, file_path, pattern type (e.g. "OpenAI key"), env_var used as replacement. Agent 5 verifies every entry was actually replaced.
SECURITY_REPORT.md Agent 5 13-section security report: executive summary, review scope+tier, critical findings, high findings, medium/low findings, platform security matrix, dependency audit results, security test results, compliance/privacy status, release integrity status, launch gate decision, required fix list, HANDOFF v5 update summary.
HANDOFF.json (v5) Agent 5 v5 HANDOFF with security_status (APPROVE/CONDITIONAL/BLOCK), findings summary, accepted_risks list.

Verification Artifacts (project directory)

Path: [project]/artifacts/verification/ — written by Agent 6.

FileContents
VERIFICATION_REPORT.md 17-section verification report: executive summary (with repair rate and top slow tasks), platform scope, spec compliance, code quality gates, test coverage + 3A coverage verification, feature parity, 6A web design contract, 6B component catalog, 6C contract conformance (static), 6D runtime API contract testing, accessibility, performance, signing & artifacts, store readiness, CI/CD, security handoff, review rubric scores (10 dimensions), overall verdict + required fixes.
screenshots/baselines/ Visual regression baselines — PNG files captured by Playwright at 4 breakpoints: desktop-1440, tablet-1024, mobile-768, mobile-390.
screenshots/diffs/ Visual regression diff images — generated when pnpm test:visual detects a deviation from baseline. CRITICAL failures = layout broken; MINOR = spacing drift.
contract_results.json Runtime API contract test output from contract_test.py: endpoints tested, violations (method, path, expected status, actual status, missing fields).
PERF_REPORT.md Performance budget test results: bundle size vs budget, API p95 response times vs budget, Lighthouse LCP/CLS scores.
PR_BODY.md Auto-generated pull request body from commit_by_task.py: structured changelog grouped by task type (feat/fix/chore), with commit hashes per task.
HANDOFF.json (v6) v6 HANDOFF: all previous fields plus coverage_pct, coverage_verdict, visual_regression_status, storybook_build, catalog_coverage, contract_conformance, runtime_contract_testing, perf_bundle, perf_api, perf_lcp, perf_cls, container_verification, final_verdict.

Critique Artifacts (project directory)

Path: [project]/artifacts/critique/ — written by Agent 4.

FileContents
CRITICISM.md 22-section critique report: executive summary (scorecard verdict + mean score), 12-Dimension Product Scorecard (D1–D12), platform assessments, critical flaws, risks, top 10 fix-first list, stop-doing list, keep-doing list, product & growth gaps, platform-specific critique, cross-platform consistency, monetization architecture assessment, UI/UX design contract assessment, what you're avoiding, improvement examples, stress question, assumption stress test, review categories A–L, per-platform verdict, overall verdict, one next action, audit checklist.
HANDOFF.json (v4) v4 HANDOFF with critic_verdict (SHIP_READY/CONDITIONAL/HOLD/RESTYLE/REDESIGN/KILL/PAUSE), scorecard_mean, dimensions_at_risk list, market_context findings.

Telemetry Artifacts (project directory)

Path: [project]/artifacts/telemetry/ — accumulated throughout the build.

FileContents
tasks.jsonl One JSON line per completed task: agent, task_id, task_name, status, task_type, platform, started_at, completed_at, duration_seconds, files_touched, tests_added, tc_required, tc_passed, repair_attempts, failure_mode, strategy, coverage_pct, secret_blocks, gates_run, gates_failed, visual_baselines, cross_task_regression, final_status, blockers. Also includes context compression events.
session_tokens.jsonl One JSON line per agent turn: timestamp, agent, task, model, input_tokens, output_tokens, cache_read_tokens, cache_write_tokens, turn_cost_usd. Consumed by token_tracker.py budget for the !budget display.
batch_results.json Output from parallel_runner.go: per-task status, exit code, stdout, stderr, duration_seconds for parallel batch execution.

.agent/ Directory (project directory)

Path: [project]/.agent/ — created during scaffold task from templates in /home/user/codesleuth/templates/.agent/.

FileContents
DONE_CRITERIA.md Project-specific definition of done checklist. Builder reads at activation. Verifier checks every [REQUIRED] and [IF_APPLICABLE] criterion before SHIP verdict.
REVIEW_RUBRIC.md 10-dimension scoring rubric (R1–R10, scored 1–5). Verifier completes before final verdict. Any dimension < 3 = HOLD.
COMMANDS.md Project-specific command reference — which scripts exist and how to run them. Builder reads this at activation; overrides defaults from agent file for this project.
ASSUMPTIONS.md Assumption ledger: A-NNN entries with statement, risk, owner (Planning/Builder), and status (Unconfirmed/Confirmed). Every inference not in the spec is recorded here.
SCOPE.md In-scope / out-of-scope / V2 feature list derived from feature-spec.md. Builder checks before implementing any feature.
FAILURE_REPORT_TEMPLATE.md Template for structured failure reports during repair mode: Task, Expected, Actual, Command Output, Failure Mode, Repair Hypothesis.
KNOWN_ERRORS.md Project-local error registry (supplements the pipeline-level KNOWN_ERRORS.md in the agents directory). Accumulates errors specific to this project's stack during the build phase.
PERF_BUDGET.json Performance budget configuration: total JS bundle size limit, largest chunk limit, API p95 response time targets per endpoint, Lighthouse LCP and CLS targets.
