🔧 Multi-Tool Support

CLI Integrations

CodeSleuth works natively with 13 AI coding tools. The pipeline rules are maintained in a .cli/ directory and synced to each tool's expected instruction format automatically. The same pipeline logic runs whether you're using Claude Code, Cursor, GitHub Copilot, Cline, Continue, or any other supported tool.

Auto-Detection & Install

The detect-and-install.mjs script detects the active AI tool and installs only that tool's instruction files:

# Auto-detect the active tool and install its rules
node .cli/sync/detect-and-install.mjs

# Force install for a specific tool
node .cli/sync/detect-and-install.mjs --tool cursor

# Dry run (show what would be installed without writing files)
node .cli/sync/detect-and-install.mjs --dry-run

# Install all tools at once (full sync)
bash scripts/sync-agent-rules.sh

Detection order (first match wins)

  1. --tool <name> flag (explicit override)
  2. Environment variables (e.g. CLAUDE_CODE, ANTHROPIC_MODEL, CURSOR_SESSION_ID)
  3. Config/marker files present in repo (e.g. .claude/settings.json, .cursor/)
  4. Process name in parent process chain
  5. Default: install all tools (full sync)

Tool-Specific Notes

Cursor

Cursor uses MDC (Markdown with Cursor metadata) rule files. CodeSleuth installs four rules into .cursor/rules/:

FileContentApply Mode
00-codesleuth-core.mdcCore identity, Zero Bypass Rule, gate enforcementalwaysApply: true
01-pipeline-gates.mdcGate protocol, activation phrases, standalone requirementalwaysApply: true
02-builder-rules.mdcRARV cycle, staged write, secret scanning, test contract enforcementAuto on Builder activation
03-security-verification.mdc20-domain security review, SHIP/NO-SHIP criteriaAuto on Security activation

GitHub Copilot

Copilot receives three instruction files in .github/instructions/:

  • codesleuth.instructions.md — core pipeline rules and gate protocol
  • security.instructions.md — security review rules and domain checklist
  • testing.instructions.md — test contract enforcement, coverage requirements, RARV cycle

Cline

Cline uses numbered rule files in .clinerules/ — loaded in order:

  • 01-codesleuth-core.md — Zero Bypass Rule and agent table
  • 02-pipeline-gates.md — gate commands and standalone requirement
  • 03-builder.md — implementation rules
  • 04-security.md — security review rules

Kiro

Kiro uses "steering" files in .kiro/steering/ — topically organized:

  • product.md — what CodeSleuth builds, platform scope, agent roles
  • pipeline-gates.md — gate command protocol
  • tech.md — technical standards (TDD, testing, contracts)
  • security.md — security review and secret scanning
  • structure.md — project isolation rule, artifact namespaces

Continue

Continue reads all .md files in .continue/rules/ as always-active context rules — no special frontmatter or metadata required. CodeSleuth installs four numbered rule files:

  • 01-codesleuth-core.md — Zero Bypass Rule, two-path activation (registered project resume + explicit Start agent N), agent table (0–7), project isolation
  • 02-pipeline-gates.md — Human gate commands, maintenance gate commands (!m-build, !m-status, etc.), sub-commands, standalone requirement
  • 03-builder.md — Per-task write pipeline, DONE criteria, TDD flow, build completion registration call
  • 04-security.md — Secret scanning patterns, SAST, absolute security rules

Detection: .continue/ directory presence, CONTINUE_SESSION_ID env var, or continue process name.

OpenCode

OpenCode receives AGENTS.md for rules and opencode.json for tool configuration. The JSON file configures the model, context settings, and tool permissions.

Syncing Rules Across All Tools

# Full sync — updates all tool-specific files from .cli/shared/ sources
bash scripts/sync-agent-rules.sh

# Verify all tool files match their source
node .cli/sync/verify-cli-instructions.mjs

# Auto-detect and install for the current environment
node .cli/sync/detect-and-install.mjs
⚠️
Edit the source, not the target

Always edit files in .cli/shared/ (or for Claude Code, CLAUDE.md), then run the sync script. Editing tool-specific files directly (e.g. .cursor/rules/) will be overwritten on the next sync.

MANIFEST.json

The .cli/MANIFEST.json is the authoritative source of truth for all tool configurations. It defines:

  • Tool identifiers and display names
  • The official instruction format for each tool
  • Source and target paths for sync
  • Detection signals (env vars, config files, process names)
  • Sync strategy: preserve (never overwrite — Claude Code's CLAUDE.md) or merge or overwrite
{
  "version": "3.0.0",
  "tools": [
    {
      "tool": "cursor",
      "displayName": "Cursor CLI / Cursor Agent",
      "officialInstructionFormat": ".cursor/rules/*.mdc",
      "sourcePath": ".cli/cursor/rules/",
      "targetPaths": [".cursor/rules/"],
      "syncStrategy": "overwrite",
      "detectionSignals": {
        "env": ["CURSOR_SESSION_ID", "CURSOR_TRACE_ID"],
        "files": [".cursor/"],
        "process": ["cursor"]
      }
    }
    ...
  ]
}
