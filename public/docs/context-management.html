🧠 Context & Compression

Context Management

CodeSleuth proactively manages the model's context window throughout a build. Instead of reacting to overflow, the Orchestrator compresses earlier history at 70% usage — ensuring agents always have room to complete the current task and produce clean handoffs.

Rolling Compression

Context compression is triggered proactively at 70% usage — not at 95% when it's too late to compress cleanly. This threshold ensures there's always room to complete the current task and generate the compressed summary.

Compression trigger

When context usage reaches 70%, the Orchestrator's compression manager:

  1. Identifies message history older than the last 2 completed tasks
  2. Summarizes each completed task's full exchange into a one-line entry:
    TASK-NNN [DONE]: [what was built, key decisions made]
  3. Replaces the full exchange with the summary in the active context window
  4. Archives the full exchange to artifacts/build/CONTEXT_ARCHIVE.md in the project directory — content is never deleted, only moved out of active context
  5. Logs a compression event to artifacts/telemetry/tasks.jsonl:
    {"event":"context_compression","tokens_freed":N,"tasks_compressed":["TASK-NNN"]}
✅
Content is never lost

Full exchange content is always archived to CONTEXT_ARCHIVE.md before compression. The archive is a permanent project artifact — the summary is what disappears from active context, not the underlying work.

Between-Agent Compression

At every gate transition, the Orchestrator compresses the previous agent's history before handing off to the next agent. Each new agent starts with a clean, focused context window.

GateCompression event
Discovery → Planning (!plan)Discovery history compressed; Planning starts fresh with feature-spec.md and HANDOFF.json
Planning → Builder (!build)Planning history compressed; Builder starts with TDD.md, INTERFACES.md, TASK-GRAPH.md
Builder → Critic (automatic)Build history compressed; Critic starts with HANDOFF v3 and built codebase reference
Critic → Security (!security)Critic history compressed; Security starts with CRITICISM.md and HANDOFF v4
Security → Verifier (!verify)Security history compressed; Verifier starts with SECURITY_REPORT.md and HANDOFF v5

Context Budget Fields in HANDOFF

The Orchestrator maintains a context_budget object in HANDOFF that every agent reads before composing its prompt:

{
  "context_budget": {
    "total_tokens": 200000,
    "reserved_pipeline_state": 8000,
    "reserved_current_task": 40000,
    "available_history": 152000,
    "compression_threshold": 0.70,
    "compressed_tasks": ["TASK-001", "TASK-002", "TASK-003"]
  }
}

Builder Context Loading Rules

The Builder never loads everything — it follows strict per-task context rules to conserve the window:

Always loaded (every task)

  • HANDOFF (active version)
  • CHECKPOINT.md
  • Current task entry from TASK-GRAPH.md
  • Relevant INTERFACES.md sections (only the module(s) the task touches)
  • COMPONENT_MAP.md (loaded once at build start)
  • .codeintel/FILE_MAP.md (loaded once at build start on existing projects)

Loaded selectively by task type

Task typeExtra files to load
Routes / pages / navigation.codeintel/ROUTES.md
API handlers / controllers.codeintel/API_ENDPOINTS.md
Database / schema / migrations.codeintel/DATABASE_SCHEMA.md
Components / UI.codeintel/COMPONENT_INDEX.md
Tests.codeintel/TEST_INDEX.md
Environment / secrets / config.codeintel/ENV_VARS.md
Scaffold (Task 1)Run pnpm codeintel after scaffold completes; load FILE_MAP.md

Never loaded during a task

  • Full TDD.md (unless specifically needed)
  • Full INTERFACES.md (only relevant sections)
  • Modules unrelated to the current task
  • Raw source directories when a codeintel map covers the same ground
  • Old conversation context (use CHECKPOINT.md summaries instead)

Repo Intelligence Maps (.codeintel/)

The .codeintel/ directory contains pre-generated structural summaries of the project. Agents read these maps instead of scanning raw source files — faster, cheaper on context budget, and consistent across all agents.

Symbol Index

At Planner activation, before producing any artifact, a codebase symbol index scan runs:

# Universal ctags (preferred)
ctags --output-format=json -R . --fields=+n > .ctags_raw.json

# Language-specific fallbacks when ctags unavailable
go doc ./...           # Go
python -m ast ...      # Python
grep -rn "^export\s" src/  # JS/TS

Produces artifacts/build/SYMBOL_INDEX.md: a structured map of file path → exported symbols → signatures. Tasks in TASK-GRAPH are annotated with reuse_existing: [SymbolName] where applicable — the Builder is told what already exists and must reuse it.

Semantic Codebase Search

For architecture-class tasks (task_type: api | auth | data | infra), the Builder uses semantic search to avoid duplicating existing implementations. The active backend is set during setup and recorded in config/context.json under active.semantic_search. The Builder reads this field at activation to determine which search path to use.

JSON index backend (active.semantic_search: "json")

Keyword-based search. No extra packages required.

# Build the index after scaffold task (first run)
python scripts/semantic_search.py index \
    --project-dir . \
    --output artifacts/build/SEMANTIC_INDEX.json

# Search before writing anything new
python scripts/semantic_search.py search \
    --query "user authentication JWT middleware" \
    --index artifacts/build/SEMANTIC_INDEX.json \
    --top 5

# Update the index after every file write
python scripts/semantic_search.py update \
    --file path/to/changed/file \
    --index artifacts/build/SEMANTIC_INDEX.json

Chroma local backend (active.semantic_search: "chroma_local")

Meaning-based vector search using ChromaDB. Setup installs chromadb automatically. On systems with an externally-managed Python (Debian, Ubuntu, etc.), it is installed into .venv — the path is recorded in active.venv in config/context.json. Use the venv's Python when invoking the script directly: 

# If active.venv is set, use its Python binary
.venv/bin/python scripts/semantic_search.py index --project-dir . --backend chroma
.venv/bin/python scripts/semantic_search.py search --query "JWT middleware" --backend chroma --top 5
.venv/bin/python scripts/semantic_search.py update --file path/to/changed/file --backend chroma

# If packages are in the system Python (active.venv is null)
python3 scripts/semantic_search.py index --project-dir . --backend chroma
💡
Backend is resolved at setup time

Agents read active.semantic_search from config/context.json and select the correct invocation automatically — you do not need to pass --backend manually during a pipeline run. The commands above are for direct use outside the pipeline (e.g. debugging, manual index rebuilds).

If matching chunks are returned from the search, the Builder reads them and reuses the existing implementation instead of writing a duplicate.
