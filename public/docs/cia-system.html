🔍 Discovery — Agent 1

Capability Intelligence Analysis (CIA)

CIA is the mechanism that transforms user Q&A into a product specification that reflects the full capability landscape. It fires at four defined trigger points — and inline whenever a user statement implies a specific integration need. CIA ensures the spec includes everything available, not just what the user thought to ask for.

Reference Catalogs

At activation, the Discovery agent loads three catalogs as internal working memory. They are never shown to the user in full — only matched findings are surfaced:

CatalogPathContentsUsed For
available_apis.md catalogs/available_apis.md 47 categories of public APIs — auth, payments, data, maps, ML, security, social, and more Identifying external data sources and service integrations the product can use
ai_skills_repositories.md catalogs/ai_skills_repositories.md Curated skill packages for PM, engineering, security, design, research, and domain-specific workflows Identifying reusable instruction-level capabilities that accelerate specific features
mcp_server-repos.md catalogs/mcp_server-repos.md MCP server directories and high-signal servers for databases, GitHub, browser automation, docs, cloud, memory Identifying live tool connections the product's development pipeline and runtime can leverage

CIA Trigger Points

CIA fires automatically at four defined moments during discovery:

TriggerWhen It FiresFocus
CIA-1: Domain Scan After Phase 2 exit (problem + outcome confirmed) — or immediately after a Phase 0 braindump Broad catalog scan based on problem domain and user type
CIA-2: Flow Scan After Phase 3 exit (core user flow documented) APIs and MCPs that enable specific flow steps (maps, payments, auth, messaging, etc.)
CIA-3: Integration Scan After Phase 9 exit (API/business logic captured) Deep API and MCP match against stated external integrations and endpoints — highest-signal trigger
CIA-4: Pre-Spec Final Scan Immediately before !spec compilation Catch-all: any resource not yet surfaced that has clear relevance

CIA Process (at each trigger point)

  1. Collect all Spec State accumulated so far (problem domain, user type, features, flows, data, integrations, monetization, platform)
  2. Scan catalogs/available_apis.md — identify APIs whose categories match the product context
  3. Scan catalogs/ai_skills_repositories.md — identify skill packages whose domain matches the product type or technical stack
  4. Scan catalogs/mcp_server-repos.md — identify MCP servers that connect to data sources or systems the product needs
  5. Filter results: keep only matches with clear, specific relevance. Discard weak or speculative matches.
  6. Present findings using the CIA Finding Format (maximum 8 findings per block)
  7. Ask one decision question: "Which of these would you like to include in the spec?"

CIA Finding Format

Every finding in a CIA block uses this exact structure:

**[RESOURCE TYPE: API / Skill / MCP] — [Resource Name]**

| Field | Detail |
|---|---|
| What it is | One sentence describing the resource |
| Why it's relevant | Specific connection to this product's stated problem, feature, flow, or user type |
| How it improves the product | Concrete capability or quality improvement it enables |
| Recommendation | INCLUDE / CONSIDER / OPTIONAL — with one-line rationale |
| Risks & Requirements | Auth method, API key cost, rate limits, setup complexity, legal/compliance |

Recommendation definitions

LabelMeaning
INCLUDE High relevance. Not including it would create obvious gaps or require building something that already exists.
CONSIDER Solid relevance. Adds real value but is not a gap risk if skipped.
OPTIONAL Low relevance. Worth noting but genuinely optional.

CIA Hard Rules

  1. Maximum 8 findings per CIA block. Priority order: INCLUDE, then CONSIDER, then OPTIONAL. When more exist, note how many were filtered.
  2. Never repeat a finding across CIA blocks. Once surfaced, tracked in Spec State — if the user already decided, do not re-surface it.
  3. Never invent resources. Every finding must come from one of the three reference catalog files.
  4. If zero relevant resources are found: output CIA-N complete — no additional capability matches found for the current product context. Do not pad with irrelevant results.
  5. Do not surface security scanning tools (GitGuardian, VirusTotal) as product features — those are pipeline tools for Agent 5, not product capabilities.

Capability Intercepts — Inline CIA

An intercept fires immediately when user input contains an explicit or strongly implied need for a specific resource — regardless of which CIA trigger point is next scheduled. This is the mechanism that ensures no explicit integration signal is ever deferred or missed.

Intercept triggers

  • User explicitly names a third-party service: "we need Stripe," "use Twilio for SMS," "hook into Salesforce"
  • User describes a single-match need: "send emails," "show a map," "OCR documents," "smart recommendations"
  • Braindump content contains integration keywords, named services, or data-source references
  • Any user answer mentions an external system, platform API, or workflow that maps to a catalog entry

Intercept format

⚡ Capability Intercept — Stripe (API)
Full-stack payment processing — matches your stated need for accepting payments in the booking flow.
Recommendation: INCLUDE
Risks: API key required; 2.9% + $0.30 per transaction; PCI compliance scope reduced but present

Include this in the spec? (yes / no / tell me more)

Intercept rules

  • One intercept per response maximum. If multiple matches fire simultaneously, bundle only those directly related (e.g., Stripe API + Stripe MCP). Unrelated matches queue for the next CIA block.
  • After the user answers, record the decision in capability_intelligence Spec State and continue discovery normally.
  • An intercepted resource is never re-surfaced at a later CIA block.
  • Intercepts do not replace scheduled CIA-1 through CIA-4 blocks — they supplement the scheduled scans by catching signals between trigger points.
  • If the user ignores an intercept, re-surface it at the next CIA block under "unresolved intercepts."
  • Intercepts are non-blocking — present the finding, ask the single approval question, wait for the user's answer before continuing.

Status values: pending (surfaced, awaiting decision), included (user approved), skipped (user declined), consider (user deferred).

Fast-Track CIA

When the initial kickoff or braindump is already detailed enough to skip most discovery phases, Fast-Track CIA compresses the schedule:

  • CIA-1 and CIA-2 run together as a single "Fast-Track CIA" block (covering resources not already handled by intercepts during braindump parsing)
  • CIA-3 runs after any API/integration clarifying questions
  • CIA-4 still runs before !spec — never skipped

Market Validation Layer

After Phase 2 exit (problem + outcome confirmed), the Discovery agent runs an automated market validation check before advancing to CIA-1:

  1. Brave Search MCP: search the stated problem. Find 2–3 evidence sources showing real demand.
  2. Firecrawl MCP: identify named competitors and scrape their pricing/feature pages.
  3. Reddit API: search relevant subreddits for pain-point threads matching the stated problem.
  4. Hacker News API: search "Ask HN" for comparable products or problems.
  5. World Bank / Data USA: pull public market sizing data for regulated or global products.

Results are recorded in Spec State under market_validation and surface as: named competitors found, evidence the problem is real, and any signals that contradict the stated differentiation.
