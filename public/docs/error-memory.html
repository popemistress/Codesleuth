🧠 Institutional Memory

Error Memory System

The Error Memory system is a cross-project, persistent knowledge base of failure patterns. When an agent discovers a non-obvious error, it captures it so no future project wastes time repeating the same debugging path.

File Locations

⚠️
Pipeline namespace — agents directory, not project directory

Error memory files live in the agents directory (/home/user/codesleuth/artifacts/pipeline/), not the project directory. They persist across all projects. Never delete these files without user confirmation.

FilePathPurpose
KNOWN_ERRORS.md artifacts/pipeline/KNOWN_ERRORS.md Full error entries with symptom, root cause, fix, guardrail, and evidence
KNOWN_ERRORS_INDEX.md artifacts/pipeline/KNOWN_ERRORS_INDEX.md Keyword → Error ID mapping for fast grep lookup. Never read KNOWN_ERRORS.md directly — use the index first.

Lookup Protocol (Read Policy)

No agent ever reads KNOWN_ERRORS.md in full — it could be enormous. The lookup is always index-first:

  1. Extract 2–4 keywords from the current error (error code, package name, function name, symptom phrase)
  2. Grep the index using the absolute path: grep -i "<keyword>" /home/user/codesleuth/artifacts/pipeline/KNOWN_ERRORS_INDEX.md → returns matching Error IDs
  3. Read only matching entries: grep -A 30 "## <ERROR-ID>" /home/user/codesleuth/artifacts/pipeline/KNOWN_ERRORS.md
  4. Apply the Guardrail field before any other action — this prevents you from making the known-bad repair attempt
  5. Apply the Fix field as hypothesis 1 in the repair loop
✅
Project-local errors

Agents also check .agent/KNOWN_ERRORS.md in the project directory for project-specific error patterns. This file is created during the scaffold task and accumulates errors specific to the project's stack.

Adding an Error — !error-add

Triggered by: a user typing !error-add, the Builder after 3+ repair attempts on the same error, or the Verifier after a repeated gate failure.

💡
Script-based — never written by hand

KNOWN_ERRORS.md is written by scripts/error_add.py, not by agents using Edit or Write tools directly. This is the same pattern as token_tracker.py and write_log.py — a dedicated script that handles ID assignment, deduplication, and writes to the correct absolute path regardless of the agent's working directory.

Command

python /home/user/codesleuth/scripts/error_add.py \
    --symptom "prisma migrate deploy fails with P3009: migration already applied" \
    --root-cause "Shadow database out of sync with migration history after manual DB reset" \
    --fix "Run 'prisma migrate resolve --applied <name>' to mark it applied without re-running" \
    --guardrail "Do not run 'prisma migrate reset' in production — drops all data" \
    --keywords "prisma,migrate,P3009,shadow database,migration history" \
    --stack "Prisma 5 / PostgreSQL" \
    --evidence "MyProject TASK-003 2026-05-31"

Output:

✅ ERROR-001 added to KNOWN_ERRORS.md and KNOWN_ERRORS_INDEX.md
   Keywords: prisma, migrate, P3009, shadow database, migration history

What the script does

  1. Reads KNOWN_ERRORS.md to determine the next ERROR-NNN ID
  2. Runs a duplicate check — if a similar symptom already exists, it skips the write and reports the existing entry
  3. Appends the formatted entry to KNOWN_ERRORS.md at its absolute agents-directory path
  4. Appends one index row per keyword to KNOWN_ERRORS_INDEX.md

Entry format (as written to KNOWN_ERRORS.md)

## ERROR-042
**Keywords:** prisma, migrate, P3009, shadow database
**Stack:** Prisma 5 / PostgreSQL
**Symptom:** [exact error message or observable failure]
**Root cause:** [why it happens]
**Fix:** [exact resolution steps]
**Guardrail:** [what NOT to do]
**Evidence:** [project name TASK-NNN date]
**Added:** 2026-05-31

---

Index format

Each keyword gets one row appended to KNOWN_ERRORS_INDEX.md:

| prisma           | ERROR-042 |
| migrate          | ERROR-042 |
| P3009            | ERROR-042 |
| shadow database  | ERROR-042 |

Checking for Known Errors — !error-check

Use !error-check [keywords] to look up relevant entries without reading the full file:

!error-check prisma migrate P3009

Procedure

  1. Grep the index for each provided keyword
  2. Collect unique Error IDs from all matches
  3. Display matching full entries from KNOWN_ERRORS.md
  4. If no match: No known errors match these keywords.

Failure Mode Classification

Before entering the repair loop, the Builder classifies the error to select the right repair strategy:

Failure ModePattern to MatchRepair Strategy
compiler_error undefined, undeclared, syntax error, unexpected token, parse error Fix syntax first; after 2 failed attempts, consider switching library
test_assertion assertion.*failed, expected.*got, FAIL\s Re-read the task spec and TC contract; re-derive expected behavior
dependency_missing cannot find module, no such file, package.*not found Re-run bootstrap dependency install before retrying
interface_mismatch undefined symbol, does not implement, has no field Re-read INTERFACES.md and SYMBOL_INDEX.md in full before retrying
timeout context deadline, timeout, timed out Task is too large — split into smaller sub-tasks; escalate to Orchestrator
permission_denied permission denied, operation not permitted Stop and escalate to Orchestrator — requires human intervention

Repair Loop

When a gate fails, the Builder follows a structured 5-attempt repair loop — not random code changes:

  1. File a Failure Report: copy .agent/FAILURE_REPORT_TEMPLATE.md, fill in Task, Expected, Actual, and exact command output. Writing the report is the first repair step — it forces structured diagnosis.
  2. Check KNOWN_ERRORS first: run the index lookup before any other action. If a match is found, apply the Guardrail and Fix.
  3. Classify the failure mode using the table above.
  4. Form one fix hypothesis informed by KNOWN_ERRORS and the classified strategy.
  5. Apply the smallest fix consistent with the strategy.
  6. Re-run targeted gate.
  7. Repeat up to 5 attempts. On attempt 3+, if the same failure mode recurs, switch to the next strategy in the table.
  8. At attempt 5, set status to BLOCKED and report with recovery details including the classified failure mode.

After repair — capture new errors

If the repair required 3+ attempts and the root cause was non-obvious:

  1. Run !error-add to trigger the Orchestrator's entry formatting
  2. Add all new keywords to KNOWN_ERRORS_INDEX.md
  3. Check for duplicate entries first — never add an entry that's already indexed

Maintenance Rules

  • Never delete an entry without user confirmation
  • Entries that no longer apply to the current stack should be marked **Status: STALE** rather than deleted — the pattern is preserved for historical reference
  • The index must stay in sync with the main file — any entry without index rows is invisible to agents and effectively doesn't exist
  • If a fix changes (e.g. a library updates its API), update the Fix field and add the old fix under a **Previously tried (incorrect):** note — this becomes the new Guardrail
