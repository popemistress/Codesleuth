## QUALITY GATES

All four gates must pass before shipping:

| Gate | Command | Requirement |
|------|---------|-------------|
| **A: Lint** | `npm run lint` | No new warnings unless documented |

If any gate is **red** → Stop feature work immediately → Enter **Repair Mode**.

## REPAIR MODE

Triggered when any gate fails.

**Rules:**
1. Fix the **smallest possible surface area**
2. No refactors, no renames, no "while I'm here" changes
3. Re-run the failing gate first
4. Then re-run all gates
5. Exit Repair Mode only when all gates are green
6. **Track consecutive failures** — If the same error appears 3 times, escalate

### Strike Limit Protocol (Mandatory)

> **Purpose**: Prevent infinite repair loops that waste resources without progress.

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        REPAIR MODE STRIKE TRACKING                          │
├─────────────────────────────────────────────────────────────────────────────┤
│  Strike 1: First attempt to fix error                                      │
│      ↓                                                                      │
│  [Re-run gate] → Still failing with SAME error?                            │
│      ↓                                                                      │
│  Strike 2: Second attempt with different approach                          │
│      ↓                                                                      │
│  [Re-run gate] → Still failing with SAME error?                            │
│      ↓                                                                      │
│  Strike 3: Final attempt                                                   │
│      ↓                                                                      │
│  [Re-run gate] → Still failing?                                            │
│      ↓                                                                      │
│  ⛔ STOP — Request human intervention / architect review                   │
└─────────────────────────────────────────────────────────────────────────────┘
```

### Error Fingerprinting

To detect "same error", compare:
- Error message text (normalized)
- File and line number (if available)
- Error code/type

If **80%+ match** on these factors, it's the same error.

## DISCIPLINE RULES

### Diff Discipline
- Prefer **small diffs** and vertical slices (end-to-end but minimal)
- Avoid refactors unless explicitly requested or required to fix a gate
- Do not rename files or reorganize directories unless asked

### Dependency Discipline
- Do **not** add new dependencies by default
- Only add if: (1) current stack cannot solve it, (2) you explain why, (3) added minimally
- If proposing a dependency: provide 1-2 no-dependency alternatives first

### Forbidden Shortcuts
These require **explicit approval** and a comment explaining why:
- `any` type
- `@ts-ignore` / `@ts-nocheck`
- Disabling lint rules globally

---

## SECURITY NON-NEGOTIABLES

- ❌ Never hardcode secrets (API keys, passwords, tokens)
- ❌ Never log secrets
- ✅ Validate all external inputs at the boundary
- ✅ Fail closed (deny by default) for auth/permissions
- ❌ Do not weaken auth, CORS, CSRF, or permission checks "to make it work"

**If touching auth/permissions:**
1. Explicitly state the access rule (who can do what)
2. Include at least one negative test or verification step
